# 架构文档：RobotCloud 生产部署架构

本文简要描述 RobotCloud 当前的线上部署形态：**Web 栈在云服务器（Volc），GPU Agent 在独立的 GPU 节点（H20）**，两者通过 SSH 隧道打通。

> 本文为架构说明，不含密钥、IP、端口映射等敏感信息。具体连接方式见 `make info-volc` / `make info-gpu`，部署步骤见 `claw/sop/`。

---

## 1. 总览

系统由两类机器组成：

- **Web 节点（Volc 云服务器）**：对外提供网站，跑前端、后端、调度器、反向代理。有公网域名，有 systemd。
- **GPU 节点（H20）**：跑 GPU Agent，执行 lerobot 训练 / 推理。是云厂商 **NAT 容器**，没有 systemd，对外只暴露一个 SSH 端口，业务端口无公网入口。

核心约束决定了架构形态：

| 约束 | 影响 |
|---|---|
| GPU 节点在 NAT 后，业务端口不可公网直连 | Web → Agent 必须走 **SSH 隧道**，不能直连 IP |
| 数据集上传不能占用 Web 节点磁盘（空间紧张） | 浏览器上传 **流式直传** GPU 节点，Web 端零落盘 |
| GPU 节点 CN 网络受限（HF 被墙、root 盘满） | 依赖装在大数据盘，HF 走镜像 |

## 2. 拓扑

```text

                          │                                                                            │
   浏览器 ──── HTTPS ────►│  nginx (443)                                                               │
                          │    ├── /            → 前端静态文件 (Next.js export)                         │
                          │    ├── /api/        → 后端 gunicorn (本机回环)                              │
                          │    ├── /storage/    → 后端静态资源                                          │

                          │                                                                  │          │

                          │         │  把"本机回环端口"转发到 GPU 节点内部 Agent 端口         │          │
                          └─────────┼───────────────────────────────────────────────────────┼──────────┘
                                    │  SSH (-L 本地端口转发)                                  │
                                    ▼                                                         ▼
                          ┌───────────────────── GPU 节点 (H20, NAT 容器) ──────────────────────────────┐

                          │  数据 / 依赖 / venv 全部落在大数据盘                                          │
                          └────────────────────────────────────────────────────────────────────────────┘
                                    ▲
                                    └──── Agent 主动 HTTPS 出网：向后端注册 + 心跳 + 上传完成回调
```

## 3. 三条数据通路

整个系统的关键就是 GPU 节点 NAT 之后，三个方向各自怎么连通：

1. **Agent → 后端**（注册 / 心跳 / 回调）
   GPU 节点**主动出网**，访问 Web 节点的公网 `https://.../api/v1`。出方向无障碍，无需任何隧道。

2. **后端 → Agent**（调度器派发训练 / 推理任务）
   GPU 节点业务端口无公网入口，所以 Web 节点用一条 **常驻 SSH 隧道（本地端口转发 `-L`）**，把"Web 本机的一个回环端口"映射到"GPU 节点内部的 Agent 端口"。
   Agent 注册时上报的地址就是 `127.0.0.1:<回环端口>`，调度器据此派发，请求经隧道进入 GPU 节点。

3. **浏览器 → Agent**（数据集直传，不经 Web 磁盘）
   nginx 上有一个 `/agent/` location，开启**流式代理**（关闭 request buffering、放开 body 大小限制），把上传请求经同一条 SSH 隧道直接转发给 Agent。
   数据**不在 Web 节点落盘**，直接流到 GPU 节点的数据盘——满足"上传不占用 Web 空间"的约束。

## 4. 各节点职责

### Web 节点（Volc）
- **前端**：Next.js 静态导出（`output: "export"`），nginx 直接托管；API 地址在**构建期**注入。
- **后端**：Django + DRF，gunicorn 监听本机回环，由 nginx 反代。
- **调度器**：独立常驻进程，负责把训练 / 推理任务派发给 GPU 节点。**必须运行**，否则任务不下发。
- **SSH 隧道**：systemd 常驻单元，维持到 GPU 节点的端口转发。
- **数据库**：SQLite（单机够用）。
- 进程用 **systemd** 管理（backend / scheduler / 隧道 / nginx）。

### GPU 节点（H20）
- **GPU Agent**：自写的轻量 HTTP 服务，接收派发，拉起 lerobot 训练 / 推理子进程，回报状态与日志。
- **lerobot 运行环境**：装在大数据盘上的独立 venv（详见 SOP 中的安装约束，如 CUDA wheel 选型、HF 镜像、torchcodec 版本对齐等）。
- 容器**没有 systemd**，Agent 以 `setsid` + while 重启循环常驻——**抗断连，但不抗容器重启**；重启后需手动拉起 Agent 脚本。

## 5. 一次完整请求的生命周期（以训练为例）

1. 用户在前端登录，**上传数据集** → 浏览器经 nginx `/agent/` 流式直传 GPU 节点，Web 端不落盘 → Agent 在数据盘落盘并回调后端标记 `ready`。
2. 用户**发起训练**（选择模型类型，如 ACT / DiffusionPolicy / Pi0 / Pi0.5 / SmolVLA）→ 后端建任务 → 调度器选中在线 GPU 节点，经 SSH 隧道 POST 给 Agent。
3. Agent 拉起 `lerobot-train` 子进程，训练完成保存 checkpoint，回报 `completed`。
4. 用户**部署 / 推理** → 调度器经隧道通知 Agent 启动推理服务，加载 checkpoint 提供推理。

> 模型类型在前端下拉框定义，后端在派发前做名称归一（如 `Pi0.5` → lerobot 的 `pi05`）。

---

## 6. 设计要点小结

- **不依赖 GPU 节点有公网入口**：所有"进 GPU 节点"的流量都收敛到一条 SSH 隧道，唯一前提是 GPU 节点能 SSH 登录、能主动出网。
- **Web 节点零数据落盘**：大文件靠 nginx 流式代理穿透，Web 磁盘只存元数据 / SQLite。
- **出方向优先**：注册、心跳、回调都让 GPU 节点主动发起，绕开入站限制。
- **可移植**：换一台 GPU 节点，只要改隧道目标 + Agent 配置即可，Web 栈不动。
